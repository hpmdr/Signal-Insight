# Signal Insight 开发心得

> 一款 Android 蜂窝网络信号监测工具的完整开发实录。
> 技术栈：Kotlin + Jetpack Compose + Material 3 + Navigation Compose

---

## 一、架构演进：从手动路由到 Navigation Compose

### 问题

最开始 App 只有信号页，页面切换用 `when(selectedDestination)` + `ExplainerRoute?` 两套手动状态管理，每新增一个页面就要改 4 个地方：枚举值、when 分发、AnimatedContent、回调类型。

### 方案

迁移到 Jetpack Navigation Compose（`NavHost` + `NavController`）：

```kotlin
NavHost(navController, startDestination = "cellular") {
    composable("cellular", enterTransition = { fadeIn(250) }) { CellularPage(...) }
    composable("settings",  enterTransition = { fadeIn(250) }) { SettingsScreen(...) }
    composable("explainer/{metricKey}", /* 带滑入/滑出动画 */ ) { ... }
}
```

### 教训

- 不要自己造路由轮子。Navigation Compose 是官方方案，动画、返回栈、参数传递全部内置
- 新增页面 = 一行 `composable()`，无需手动管理状态
- 迁移工程量中等（改 MainScreen 核心路由逻辑），但收益长远

## 二、MIUI 兼容：LTE SINR 读取

### 问题

MIUI/Xiaomi 设备上，`TelephonyCallback.CellInfoListener` 回调的 `CellSignalStrengthLte.rssnr` 全部返回 `Int.MAX_VALUE`，导致 4G LTE 的 SINR 始终显示 N/A。但 Cellular-Z 等同类 App 能正常读取。

### 调试过程

1. 通过 ADB logcat 确认 CellInfo 确实返回 MAX_VALUE
2. `dumpsys telephony.registry` 检查系统底层 SignalStrength 数据，发现真实值存在
3. 确认 SignalStrength 回调和 CellInfo 回调走的是两条不同的 API 路径

### 解决方案

双监听器模式：

```kotlin
// 主数据源：CellInfo 回调（含 PCI、RSRP 等全部信息）
tm.registerTelephonyCallback(executor, cellInfoCallback)

// SINR 备用源：PhoneStateListener（MIUI 上能读到真实 RSSNR）
@Suppress("DEPRECATION")
val ssListener = object : PhoneStateListener(executor) {
    override fun onSignalStrengthsChanged(signalStrength: SignalStrength?) {
        // 提取 LTE RSSNR 存入 _lteSinrFallback
    }
}
tm.listen(ssListener, LISTEN_SIGNAL_STRENGTHS)
```

### 关键坑

1. **`PhoneStateListener()` 不能在后台线程创建** —— 无参构造器依赖当前线程的 Looper。在 `Dispatchers.IO` 上会抛 NullPointerException。必须传入 `Executor`（API 31+）或 `Looper`。
2. **备用值不能用于邻小区** —— SignalStrength 回调返回的是设备级聚合值，不带 PCI 标识，无法区分各邻小区。如果强行使用，所有邻小区 SINR 都会显示相同的值。
3. **只在服务小区使用备用值**，邻小区显示 N/A（至少不会误导）。

### 最终策略

三级降级：`CellInfo.rssnr → _lteSinrFallback → Int.MAX_VALUE`。前两者均无效才算不可用。

## 三、组件选择：为什么不用 NavigationBar 做 SIM 切换

### 最初做法

用 `NavigationBar` 组件做底部 SIM 卡切换（SIM1/SIM2），因为它看起来"像底部导航栏"。

### 问题

1. **语义错误** —— NavigationBar 的官方定义是"App 顶层目的地导航"（3-5 个目的地），不是"页面内视图切换"
2. **高度浪费** —— NavigationBar 固定 80dp（56dp 内容 + 24dp 手势条 inset），SIM 切换只需要约 50dp
3. **不可写高度的设计** —— Material 3 的设计令牌（design token）固定了高度，官方不推荐覆写

### 调研结论

查阅 Material 3 规范后发现：

| 组件 | 语义 | 适用场景 |
|------|------|---------|
| NavigationBar | "去哪页" | App 顶层目的地（3-5个） |
| Tabs | "看什么" | 页面内同级内容切换 |
| **SegmentedButton** | **"选哪个"** | 视图选项切换（2-5个） |
| ConnectedButtonGroup | "选哪个" | Expressive 版本（替代 SegmentedButton） |

SIM1/SIM2 切换属于"2 个互斥视图中选一个"，正确的组件应该是 `SingleChoiceSegmentedButtonRow`。

### 最终实现

用户选择了**弹性水滴动画方案**（非标准 SegmentedButton）：

```kotlin
// 双边界独立弹簧算法
val animatedLeft by animateDpAsState(targetLeft, spring(DampingRatioMediumBouncy, StiffnessLow))
val animatedRight by animateDpAsState(targetRight, spring(DampingRatioMediumBouncy, StiffnessLow))
val currentPillWidth = animatedRight - animatedLeft  // 拉伸形变

Box(Modifier.offset(x = animatedLeft).width(currentPillWidth).clip(RoundedCornerShape(20.dp)).background(activeColor))
```

核心原理：Left 和 Right 两个边界各自独立绑定 spring。切换时领先沿（Right）先走，落后沿（Left）因弹性滞后慢半拍才收缩 → 滑块在移动过程中被拉长 → 视觉上呈现水滴拉伸、流动、吸附的效果。

## 四、Compose 动画实践

### 页面转场

```
进入详情：详情从右滑入 + 淡入     主页向左半滑 + 淡出
返回主页：主页从左半滑 + 淡入     详情向右滑出 + 淡出
标签切换：淡入(250ms) + 淡出(150ms)
```

关键：**进入和退出必须对称**。最初退出只写了 `fadeOut(150)` 忘了 `slideOut`，导致详情页"闪一下"消失。官方推荐的是 `slideIn + fadeIn togetherWith slideOut + fadeOut`。

### 底部栏的显示时机

`AnimatedVisibility` + `fadeIn/fadeOut` 不能完美解决 Scaffold 底部栏的动画 —— Scaffold 会在 bottomBar 内容变化时重新计算 `innerPadding`，导致高度跳跃。

最终方案：放弃淡入淡出动画，直接用 `LaunchedEffect + delay(300ms)` 控制闪现时机，等页面过渡动画播完后再显示底部栏。

```kotlin
LaunchedEffect(showBottomBar) {
    if (showBottomBar) {
        delay(300) // 等页面动画播完
        delayedShow = true
    }
}
```

## 五、数据层陷阱

### StateFlow 防抖

网络切换（5G↔4G）时，系统会短暂发送空 CellInfo，导致 UI 闪烁 N/A。解决方案：

```kotlin
// 防抖保护：只有在「数据有效」或「从未有过有效数据」时才更新
if (sim1Valid || !sim1HasValidEver) {
    _sim1Data.value = sim1Data
}
```

### derivedStateOf 不能冷读 StateFlow.value

```kotlin
// ❌ 错误：derivedStateOf 只会在重组时触发，不会订阅 StateFlow
val options by remember { derivedStateOf { getOptions(sim1Data.value) } }

// ✅ 正确：先 collectAsState()
val sim1Data by viewModel.sim1SignalData.collectAsState()
val options = remember(sim1Data) { getOptions(sim1Data) }
```

### WhileSubscribed(5000)

最后一个订阅者消失后 5 秒内仍保持上游活跃。适合快速切后台/前台场景，避免反复创建数据流连接。

## 六、权限管理

### 动态权限流程

1. 启动时检查 `ACCESS_FINE_LOCATION` + `READ_PHONE_STATE`（Android 13+ 用 `READ_BASIC_PHONE_STATE`）
2. 未授权 → 显示权限引导页
3. 用户拒绝 → 检测「不再询问」，引导去系统设置
4. 从系统设置页返回后（`ON_RESUME`）重新检查权限
5. 授权后自动启动数据收集

### 关键实现

```kotlin
// MainScreen 中的生命周期监听
DisposableEffect(lifecycleOwner) {
    val observer = LifecycleEventObserver { _, event ->
        when (event) {
            ON_RESUME -> {
                permissionViewModel.checkAllPermissions(activity)
                if (allPermissionsGranted) cellularViewModel.resumeDataCollection()
            }
            ON_PAUSE -> cellularViewModel.pauseDataCollection() // 省电
        }
    }
    lifecycleOwner.lifecycle.addObserver(observer)
    onDispose { lifecycleOwner.lifecycle.removeObserver(observer) }
}
```

## 十、给后来者的建议

1. **优先使用官方组件** —— Navigation Compose、Material 3 组件都有明确的语义和使用场景。用对组件能省 80% 的心。
2. **API 兼容性要提前验证** —— MIUI 等定制 ROM 的行为与 AOSP 不完全一致，需要在目标设备上实际测试。
3. **动画对称** —— 进入动画和退出动画要配对设计，单向动画看起来会很奇怪。
4. **组件语义比视觉效果更重要** —— 用 NavigationBar 装 SIM 切换器，效果再好也改变不了它语义错误的事实。后期改造成本更高。
5. **Scaffold 的 bottomBar 动画** —— AnimatedVisibility + Scaffold 存在布局跳变问题，简单地用 delay + 闪现可能比复杂的动画效果更好。
6. **测试真实设备** —— 模拟器无法模拟 MIUI 的 CellInfo 行为，必须在真机上调试。
